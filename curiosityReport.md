This would allow an Agentic AI to have even more context on the real impact of a given change. ## What Progressive Delivery Is

**Progressive delivery** is the strategy of releasing a new version gradually to a few users, and then scale up slowly to all users over time. This helps reduce user exposure to any bugs that may crop up on the new version. 

Canary Deployment is the simple form of progressive delivery where a small amount of traffic is sent to the new version first. Like I said above, this limits the damage of a bad version with bugs or performance issues. Feature flags are a way that teams can enable certain code for specific users or environments.

A more advanced version is called a guarded rollout. In this version, metrics decide how to handle the delivery. If the service keeps meeting standards like low error rate or latency, the rollout continues and ramps up. If anything gets worse, the system can automatically roll back to the more stable version. This allows the metrics to play a larger role in the deployment process. The interesting aspect of this that i wanted to research was the ability of an Agentic AI to take those delivery metrics, and make modifications automatically based on predetermined specs and expectations.


## Tools That Already Do This

As I looked into it, I found a few technologies that already support this kind of delivery, but not in the exact way that I was looking for:

**Argo Rollouts** is a Kubernetes progressive delivery controller. Rollouts can create analyses during a canary or blue-green deployment which can use systems like Prometheus and decide whether the process should continue or abort. I think this general idea makes a lot of sense, because it creates a feedback loop that is vital for AI based tools to be effective.

**Flagger** uses a similar idea. Its documentation says it gradually shifts traffic to a canary while measuring metrics like request success rate and request duration. It can also use custom metrics and webhooks. If too many checks fail, it rolls the release back automatically.

**LaunchDarkly** is known for feature flags, but it also supports guarded rollouts and autogenerated metrics. What made it especially relevant to my topic is that its AI documentation includes metrics like duration, token usage, generation success, generation error, time to first token, and output satisfaction. This allows it to interact with AI a little more heavily.

## Machine Learning Needs That Go Beyond Normal Metrics

Normally, teams watch metrics like uptimes, latency, error rate, etc. which still matter for ML systems, but don't fully cover what is needed for a more agentic system. 

ML systems may also need to track:

- token cost
- user satisfaction
- output quality
- **data drift**
- **prediction drift**

[Evidently AI](https://docs.evidentlyai.com/) is a great example because it focuses on the evaluation and monitoring of AI powered systems. This includes drift and quality over time. This could possibly be fed back into an agentic LLM as context about the most recent delivery, or the one that is currently being rolled out.

## Example Scenario

If a company is using an agent to do a progressive product delivery, they could have the agent set up to start with 5% of traffic. Then based on the metrics that the agent recieves back from that portion of the deployment, it could rewrite or refactor areas of poor performance, present it for human review, and then push out to 5% again until it starts seeing improvements. It could continue this iterative process until the delivery is complete, fixing bugs and issues along the way.

## What I Learned

As the world becomes better at using the new tools that are available, each section of a software lifecycle is going to be automated more and more. Having live feedback and near instantaneous fixes will allow DevOps engineers to focus on even larger scale problems and make deliveries faster and more reliable. Using progressive delivery metrics to drive fixes is a very interesting loop that I think allows for quicker solutions and better deploys.
